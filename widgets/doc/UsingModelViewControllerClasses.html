Advanced Application Development: Using Model-View-Controller Architecture

Separated presentation design patterns help you work more effectively

Building an app programmatically gives significantly more control over the code structure and architecture. As apps grow in complexity, modular architecture and separation of concerns are important to achieve optimum performance, scalability, and testability. Additionally, following a standard software architecture pattern such as model-view-controller (MVC) as shown in Figure 1 enables you to build modular apps from smaller components, rather than having a single monolithic app file.
Figure 1. The model-view-controller (MVC) software architecture pattern.

Developing Model Classes

A model class has responsibilities to:
  • Store relevant data for the application
  • Store relevant states for the application
  • Provide interfaces to the algorithms needed for the application
Model classes should generally be implemented as a handle class so that multiple components can reference the same model. Widgets Toolbox provides superclasses that extend handle to provide additional capabilities. These superclasses include wt.model.BaseModel and wt.model.BaseSession. Using these is also necessary to work together with some other Widgets Toolbox classes including the view and controller superclasses that will be described later. These classes are shown below in Figure 2:
Figure 2. BaseModel and BaseSession classes UML diagram
The intended uses of these classes are:
  • The class wt.model.BaseModel is needed when you utilize the view and controller superclasses wt.abstract.BaseViewController and wt.abstract.BaseViewChart to implement modular application components.
  • The class wt.model.BaseSession, which is a subclass of wt.model.BaseModel, is needed when you use the top-level application superclasses wt.apps.BaseSingleSessionApp or wt.apps.BaseMultiSessionApp. These are described in Reusable Superclasses for Hand-Code Apps.

Examples

BaseModel

The class zooexample.model.Animal implements a model class for a zoo animal's information.

animal1.Name = "Thomas";
animal1.Species = "Tiger";
animal1.Sex = "male";
animal1.BirthDate = "June 6, 2014"

Developing View and Controller Classes

A view or controller class has several responsibilities:
  • View: Display a representation of data in the model
  • Controller: Provide a set of controls that manipulate model data or states
  • Keep the displayed content or controls synchronized with the current state of the model
View and controller classes may be implemented using the ChartContainer or ComponentContainer superclasses. However, Widgets Toolbox also provides superclasses wt.abstract.BaseViewController and wt.abstract.BaseViewChart that extend ChartContainer and ComponentContainer to provide additional capabilities. These also share a common mixin class wt.mixin.ModelObserver that provides functionality to observe changes in a model class. These classes are shown below in Figure 3:
Figure 3. BaseViewChart and BaseViewController classes UML diagram
You can determine which superclass to use for your view or controller class based on the intended contents:
  • The class wt.abstract.BaseViewChart may be used when the view/controller to be placed in the ContextualView contains only axes and can be implemented as a ChartContainer subclass. This is generally used as a view component only.
  • The class wt.abstract.BaseViewController must be used when the view/controller to be placed in the ContextualView contains any combination of axes and app components and will be implemented as a ComponentContainer subclass. This can be used for a view or controller component, or a mix of the two.
Depending on the implementation, in MATLAB it is sometimes convenient to combine view and control components together in a hybrid component. Regardless of whether you are implementing a view, controller, or hybrid view/controller component, you should use the BaseViewController class unless your component would contain only axes for charting data.

Examples

BaseViewController

The class zooexample.view.Animal implements a view/controller class for a zoo animal's information.
fig = uifigure("Position",[100 100 300 220]);
g = uigridlayout(fig,[1 1]);
view1 = zooexample.view.Animal(g);
view1.Model = animal1

ModelObserver

ModelObserver is a mixin class that is utilized by both wt.abstract.BaseViewChart and wt.abstract.BaseViewController to provide functionality to listen to state changes in a model class and notify your view or controller when an update is needed to synchronize the view to the current model state.

Description

ModelChanged
Triggered when the object within the Model proeprty has triggered a ModelChanged event, indicationg a property or state inside the model has changed.

Properties

Access
Description

abstract
Placeholder for the model class containing the data and states for the view or controller component.
The concrete class for your component must define the model class and enforce its datatype.

Methods

Name
Access
Description
constructDefaultModel
protected
newModel = constructDefaultModel(obj)
Generates a scalar object instance of the model class. It will use the same datatype as the current value in the Model property, even if it is an empty object array. This is useful in instances where you need to create a new copy of the model for the component. It is utilized by the getScalarModelToDisplay method.
For this to work, your model must be capable of being instantiated with no input arguments.
getModelClassName
protected
className = getModelClassName(obj)
Returns the class name of the Model property contents.
getScalarModelToDisplay
protected
[model, validToDisplay] = getScalarModelToDisplay(obj)
This is useful for the update method of your component. It retrieves a scalar instance of the correct model type and indicate its validity. In case obj.Model is empty, it will still return a scalar instance with default values, but the validToDisplay flag will be false.
If validToDisplay is false, typically you should disable the view or control components to prevent user interaction.
